## 2. 실행 원칙
1. 범위 고정: MVP 범위 외 기능은 구현하지 않는다.
2. 단일 진실 원천(SSOT): 요구사항 충돌 시 아래 우선순위를 따른다.
1. 기능 정책/수용기준: 기능명세서
2. 화면 동작/전환/CTA: UI/UX 명세서
3. 데이터 구조/관계/출처: ERD
3. 홈 정책 준수: 홈은 방문 유도 허브로 운영하고 일반 상품 상세 노출은 검색/상세 화면으로 분리한다.
4. 모킹 우선: 시연 안정성을 위해 MVP 핵심 플로우는 모킹 데이터 기준으로 완성한다.
5. 실패 허용 설계: AI/API 실패 시 즉시 Fallback UX로 전환한다.

## 3. 개발 범위 확정
### 3.1 MVP 인스코프
1. 소비자: 홈 허브, 검색/비교, 드랍, 장보기 리스트, 지도/추천 동선, 알림함
2. 상인: 상품 등록(AI 보조)
3. 공통: 인증/권한, 알림(인앱/푸시), 공통 에러/빈 상태

## 4. 기능-화면-데이터 매핑 (개발 기준)

## 5. 작업 분해 구조 (WBS)
### 5.1 Track A: 데이터/백엔드
1. 스키마 확정: ERD 기준 테이블/관계/enum 반영
2. Seed 구성: 공공+모킹 데이터 분리 로딩
3. API 1차: 홈/검색/드랍/장보기/동선/알림/인증
4. API 2차: 상인 등록 AI 보조 입력/결과 검토/게시
5. 실패 정책: SLA 초과/빈 데이터/외부 API 실패 공통 응답 포맷

### 5.2 Track B: 프론트엔드
1. 라우팅/탭 구조: 기본 진입 SCR-C-01, 하단 탭(홈/검색/장보기/지도/마이)
2. 공통 컴포넌트: 리스트 카드, 상태 배지, 에러/빈 상태
3. 소비자 플로우: 홈→검색/상세→드랍→알림함→장보기→동선
4. 상인 플로우: 상인 홈→상품 등록(입력/검토)
5. 관측 코드: screen_view/click_cta/api_error 이벤트 삽입

### 5.3 Track C: AI/API 연동
1. 상품 등록 AI: 이미지/음성 텍스트 입력 → 상품명/카테고리/설명 출력
2. RAG 구현: 샘플 레시피 데이터셋 인덱싱 및 MVP 응답 플로우 연결
3. Fallback 구현: 인식 실패 시 수동 입력 전환, 추천 실패 시 템플릿 응답
4. 비밀키 운영: `.env` 분리, 호출량 제한, 에러 로깅

### 5.4 Track D: QA/릴리스
1. 화면별 CTA/Navigation 점검
2. 기능별 AC 점검(Functional 기준)
3. 데이터 정책 점검(SLA/갱신시각/상태값)
4. 시연 시나리오 리허설(소비자 플로우, 상인 플로우)

## 7. 데이터 운영 및 SLA 적용
1. 가격: 24시간 내 1회 갱신
2. 재고: 하루 1~2회 갱신
3. 드랍: 이벤트 발생 시 즉시 등록
4. 사용자 노출: 모든 핵심 데이터에 최종 업데이트 시각 노출
5. 품질 기준: 가격 오차 ±10%, 재고 상태 3단계(있음/적음/품절)

## 8. API/AI 의존성 및 대체 전략
### 8.1 필수 의존성
1. 인증 API
2. 홈/검색/드랍/알림/장보기/동선 API
3. AI 인식/LLM API (상품 등록 보조)

### 8.2 장애 시 대체
1. AI 실패: 수동 입력 폼 자동 전환
2. 알림 실패: 인앱 알림 대체 + 1회 재시도
3. 외부 데이터 지연: 마지막 정상 스냅샷 + 갱신 필요 배지

## 9. 품질 게이트 (Go/No-Go)
### 9.1 기능 게이트
1. FR-C-01/02/04/05, FR-M-01, FR-A-01/02 동작 확인
2. 소비자 핵심 경로 1회 완료 가능
3. 상인 등록 경로 1회 완료 가능

### 9.2 UX 게이트
1. 모든 MVP 화면에 CTA 1개 이상 동작
2. 명세된 Navigation 전이 100% 일치
3. 로딩/빈/오류 상태 존재

### 9.3 데이터 게이트
1. P0 엔터티 시드 데이터 준비 완료
2. 모킹/공공데이터 경계 명시 완료
3. 알림 target_type 기반 라우팅 검증 완료

## 11. 변경관리 규칙
1. 범위 변경 제안은 이슈로 등록 후 승인 전 구현 금지
2. 문서 충돌 발생 시 SSOT 우선순위에 따라 24시간 내 정정
3. Screen ID, FR ID, Entity 명칭은 임의 변경 금지
4. Phase 2/3 항목을 MVP 화면에 섞어 넣지 않는다.

## 12. 리스크 및 대응
1. 문서 간 참조 불일치: 릴리스 전 매핑 테이블 재검수
2. 모킹 데이터 품질 부족: 시나리오 기준 데이터 세트 사전 검증
3. AI 응답 불안정: 프롬프트 고정 + Fallback 강제
4. 지도 정밀도 한계: 구역 텍스트 안내를 기본 병행

## 13. 착수 체크리스트
1. 개발 브랜치 및 환경변수 템플릿 생성
2. DB 마이그레이션/시드 스크립트 실행 확인
3. API 명세서(엔드포인트/응답 필드) 공유
4. 화면별 테스트 계정/테스트 데이터 배포
5. 데모 시나리오 체크리스트 배포

## 14. 완료 정의 (Definition of Done)
1. MVP 인스코프 기능이 명세된 화면 흐름대로 동작한다.
2. 문서-화면-데이터 간 매핑 충돌이 없다.
3. 실패 케이스(알림/AI/API)에서 대체 UX가 동작한다.
4. QA 체크리스트 통과 및 데모 리허설 2회 완료.
